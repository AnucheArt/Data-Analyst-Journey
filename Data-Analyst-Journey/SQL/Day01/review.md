## Сильные стороны

### 1. Хорошее понимание фильтрации

Правильно используется `WHERE` для отбора оплаченных заказов:

```sql
SELECT *
FROM orders
WHERE status = 'paid';
```

Понимание фильтрации закрепилось после первой ошибки, когда сумма была посчитана по всем заказам.

---

### 2. Уверенная работа с агрегатными функциями

Правильно применяются:

```sql
COUNT(*)
SUM(amount)
AVG(amount)
MAX(amount)
MIN(amount)
```

Особенно хорошо, что агрегатные функции используются вместе с фильтрацией:

```sql
SELECT SUM(amount)
FROM orders
WHERE status = 'paid';
```

---

### 3. Понимание `GROUP BY`

Правильно составлен сводный отчёт:

```sql
SELECT
    jobe,
    COUNT(*) AS employees,
    AVG(salary) AS avg_salary,
    MAX(salary) AS max_salary,
    MIN(salary) AS min_salary,
    SUM(salary) AS total_salary
FROM employees
GROUP BY jobe;
```

Этот запрос уже похож на реальную рабочую задачу аналитика.

---

### 4. Понимание разницы между `WHERE` и `HAVING`

Правильно объяснено:

- `WHERE` фильтрует строки до группировки;
- `HAVING` фильтрует уже сформированные группы.

Пример:

```sql
SELECT
    jobe,
    COUNT(*) AS employees
FROM employees
GROUP BY jobe
HAVING COUNT(*) > 2;
```

---

### 5. Первый корректный `JOIN`

Правильно соединены таблицы `orders` и `customers`:

```sql
SELECT
    customers.first_name,
    orders.amount
FROM orders
JOIN customers
    ON orders.customer_id = customers.customer_id;
```

Есть понимание, зачем указывать таблицу перед названием столбца:

```sql
orders.customer_id
customers.customer_id
```

Это особенно важно, когда одинаковые названия столбцов существуют в нескольких таблицах.

---

### 6. Хороший стиль оформления

Код читаемый:

- ключевые слова написаны в верхнем регистре;
- каждая часть запроса начинается с новой строки;
- используются понятные псевдонимы через `AS`;
- запросы разделены комментариями.

## Ошибки, которые были допущены

### Ошибка 1. Пропущена фильтрация оплаченных заказов

Было:

```sql
SELECT SUM(amount)
FROM orders;
```

Но задача требовала сумму только оплаченных заказов.

Исправление:

```sql
SELECT SUM(amount)
FROM orders
WHERE status = 'paid';
```

**Вывод:** перед написанием запроса нужно выделить все условия задачи.

---

### Ошибка 2. Точка с запятой поставлена раньше конца запроса

Было:

```sql
SELECT *
FROM orders
WHERE status = 'paid';
ORDER BY amount DESC
LIMIT 1;
```

Точка с запятой завершила запрос до `ORDER BY`.

Исправление:

```sql
SELECT *
FROM orders
WHERE status = 'paid'
ORDER BY amount DESC
LIMIT 1;
```

**Вывод:** `;` ставится только после последней части запроса.

---

### Ошибка 3. Неверный столбец в `GROUP BY`

Сначала использовался другой столбец вместо того, который выводился в `SELECT`.

Правило:

> Все обычные столбцы из `SELECT`, которые не находятся внутри агрегатной функции, должны присутствовать в `GROUP BY`.

---

### Ошибка 4. Обычный столбец был выведен вместе с агрегатом

Было:

```sql
SELECT salary, AVG(salary)
FROM employees
GROUP BY jobe;
```

SQL не знает, какую конкретную зарплату нужно вывести для каждой группы.

Исправление:

```sql
SELECT
    jobe,
    AVG(salary) AS avg_salary
FROM employees
GROUP BY jobe;
```

---

### Ошибка 5. Использовался `COUNT(jobe)` вместо `COUNT(*)`

```sql
COUNT(jobe)
```

считает только строки, где `jobe` не равен `NULL`.

Для подсчёта всех строк группы лучше использовать:

```sql
COUNT(*)
```

## Замечания по структуре базы

### 1. Название столбца `jobe`

Название `jobe` выглядит как опечатка.

Более профессиональные варианты:

```text
job_title
position
department
```

Если столбец содержит должность, лучше использовать:

```text
job_title
```

Если он содержит отдел:

```text
department
```

Переименование лучше сделать до того, как таблица станет большой.

---

### 2. Несоответствие `user_id` и `customer_id`

Ранее таблица `orders` создавалась со столбцом:

```sql
user_id
```

В блоке `JOIN` используется:

```sql
orders.customer_id
```

Нужно выбрать один вариант и использовать его во всей базе.

Рекомендуемый вариант для текущих таблиц:

```text
customers.customer_id
orders.customer_id
```

Если таблица `orders` уже содержит `user_id`, JOIN должен выглядеть так:

```sql
SELECT
    customers.first_name,
    orders.amount
FROM orders
JOIN customers
    ON orders.user_id = customers.customer_id;
```

## Что можно улучшить в `practice.sql`

### 1. Добавить псевдонимы таблиц

Сейчас:

```sql
SELECT
    customers.first_name,
    orders.amount
FROM orders
JOIN customers
    ON orders.customer_id = customers.customer_id;
```

Позже можно писать короче:

```sql
SELECT
    c.first_name,
    o.amount
FROM orders AS o
JOIN customers AS c
    ON o.customer_id = c.customer_id;
```

Пока полный вариант полезнее для понимания, но алиасы понадобятся в сложных запросах.

---

### 2. Добавить сортировку в сводный отчёт

Полезно видеть должности с самым большим фондом оплаты труда сверху:

```sql
SELECT
    jobe,
    COUNT(*) AS employees,
    AVG(salary) AS avg_salary,
    MAX(salary) AS max_salary,
    MIN(salary) AS min_salary,
    SUM(salary) AS total_salary
FROM employees
GROUP BY jobe
ORDER BY total_salary DESC;
```

---

### 3. Округлять средние значения

В PostgreSQL среднее может выводиться с большим количеством знаков.

Пример:

```sql
SELECT
    jobe,
    ROUND(AVG(salary), 2) AS avg_salary
FROM employees
GROUP BY jobe;
```
